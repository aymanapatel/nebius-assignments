== Data Preparation Pipeline

The notebook uses the **SST-2 Dataset** (Stanford Sentiment Treebank) for binary sentiment classification (Positive vs. Negative).

=== Text Cleaning & Tokenization
Before passing text to a model, noise must be removed to represent similar pieces of text consistently. 

[options="header", cols="1,2,2"]
|===
| Step | Action | Example
| *Lowercasing* | Converts all text to lowercase | "Good Movie" -> "good movie"
| *Punctuation* | Keeps only basic punctuation | "Wow!!!" -> "wow!!!"
| *Hyphens* | Replaces hyphens with spaces | "well-written" -> "well written"
| *Whitespace* | Collapses multiple spaces | `[space][space]word` -> ` word`
|===

=== Bag-of-Words (BoW) Representation
Because linear models require fixed-length numerical feature vectors, text is converted using BoW:

. *Vocabulary Building:* Extract the top `V=10,000` most frequent tokens from the training set.
. *Vectorization:* Convert each sentence into a sparse vector of token frequencies.

icon:warning[] *Note:* While effective for simple models, fixed-vocab BoW discards context and word order. Modern LLMs solve this with sub-word/byte-pair encodings.

== Logistic Regression & Training

=== Model Architecture
The logistic regression prediction function maps linear transformations through a Sigmoid activation:
[stem]
++++
\hat{Y} = \frac{1}{1+exp^{-(wx +b)}}
++++

.PyTorch Implementation Requirements:
* `w` (Weights) and `b` (Bias) wrapped as `nn.Parameter`.
* Initialization strategies: Zeros vs. Small Random values.
* Thresholding for binary predictions (Class 1 if stem:[p \ge 0.5]).

=== Training Loop Configuration
Training is orchestrated using *mini-batch Stochastic Gradient Descent (SGD)* with **Binary Cross-Entropy Loss (BCE)**.

.Hyperparameter Trade-offs
[options="header", cols="2,3,3"]
|===
| Hyperparameter | Low Value Effect | High Value Effect
| *Learning Rate (LR)* | Slow convergence; might get stuck in local minima. | Unstable training; divergence; skipping over minima.
| *Batch Size* | Noisier updates; slower hardware execution; acts as implicit regularization. | Smoother gradients; faster per-epoch timing; requires more memory.
|===

== L1 Regularization & Sparsity

Regularization prevents overfitting by penalizing large weights.

[options="header", cols="1,3,3"]
|===
| Regularization Type | Mathematical Penalty | Effect on Features
| *L1 (Lasso)* | stem:[\lambda \sum \|w_i\|] | Implicit feature selection. Pushes non-important weights exactly toward zero (*Sparsity*).
| *L2 (Ridge)* | stem:[\lambda \sum \|w_i\|^2] | Shrinks weights down but rarely zeroes them out. Smooth gradient scaling.
|===

icon:lightbulb[] *Proximal Descent vs. Vanilla SGD:*
Because the L1 gradient (stem:[\lambda \cdot \mathrm{sign}(w)]) is constant regardless of the weight's magnitude, vanilla SGD leaps around zero noisily at the end of training. Proximal descent explicitly thresholds small weights to exactly `0`, yielding true sparsity.

== Optimization Algorithms

Moving beyond vanilla SGD, the notebook tests optimizers on distinct loss topologies.

=== Loss Functions Analyzed
. **Convex Bowl:** stem:[f(x,y)= x^2 + 2y^2]. A simple, smooth convex environment with a single global minimum. Easy to optimize.
. **Six-Hump Camel Function:** A difficult, non-convex landscape with narrow curved valleys and multiple local/global minima.

=== Optimizer Comparison
[options="header", cols="1,3,2"]
|===
| Optimizer | Mechanism | Best Use Case
| *Gradient Descent (GD)* | Standard steepest descent using the full/batch gradient. | Simple convex problems or baseline testing. Very prone to oscillation in narrow valleys.
| *Momentum* | Accumulates a moving average of past gradients (velocity) to dampen oscillations. | Navigating ravines or steep topological drops.
| *AdaGrad* | Adapts learning rates per parameter based on historical squared gradients. | Sparse features (frequent features get lower LR, rare get higher). Con: LR vanishes over time.
| *Adam* | Combines Momentum (1st moment) and RMSProp (2nd moment) with bias correction. | The default standard for deep learning and complex non-convex landscapes. Highly stable.
|===

== Takeaways & Best Practices

. *Numerical Stability is Critical:* When implementing `log-loss` or Logits, adding a tiny `epsilon` (e.g., `1e-15`) or clipping logits prevents calculations like `log(0)`, which yield NaNs.
. *L1 for Interpretability:* If you have thousands of features (like BoW tokens) and need to know *which* words matter most, L1 regularization will trim the noise effectively.
. *Adam for Complex Topologies:* As observed in the Six-Hump Camel function, standard GD can get easily trapped in local minima or bounce endlessly in ravines. Adam's moment-based tracking allows it to gracefully navigate difficult surfaces.
